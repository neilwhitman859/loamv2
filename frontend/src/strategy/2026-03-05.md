## Strategic Recommendations

### 1. Start with AI Insights (Tier 1) — It's the Fastest Win

The insight tables are the most impactful for user experience and require zero external API integrations. We have all the input data: wine names, producers, grapes, regions, appellations, varietal categories. Claude Haiku can generate rich, knowledgeable content about all of these.

**Suggested order:**
1. `appellation_insights` (752 records — small, high-value)
2. `grape_insights` (707 records — small, high-value)
3. `producer_insights` (30,418 records — medium, batched)
4. `wine_insights` (100,440 records — large, batched by importance)
5. `region_insights`, `country_insights`, `soil_type_insights` (small, quick)

### 2. The wine_candidates.wines_id Gap Should Be Fixed

All 100,646 `wine_candidates` have `wines_id = NULL`. This means there's no direct link from the source data back to the canonical wines. We've been working around it with producer name matching, but this linkage would:
- Make future data enrichment easier (new data about a wine_candidate can flow to the wine)
- Enable re-processing if we improve the wine creation pipeline
- Support the "search candidate → wine" handoff described in the schema

**Fix:** A single SQL UPDATE matching on `(producer_name_normalized, wine_name_normalized)` would cover ~99% of candidates.

### 3. Geo-Coordinates Are a Force Multiplier

Once appellations have lat/lng:
- Weather data becomes accessible (Open-Meteo is free)
- Map visualizations become possible
- Water body proximity can be calculated
- Growing season validation becomes data-driven

This unlocks Tiers 2 and 4 simultaneously.

### 4. Consider a "MVP Enrichment" Strategy

Rather than enriching all 100K wines uniformly, consider:
- **Top 10,000 wines** (by vintage count, as a proxy for importance/popularity)
- Full enrichment: insights + scores + pricing + weather
- This gives a rich experience for the wines people actually care about
- Expand from there

### 5. The Schema Is Solid — No Changes Needed

After building and populating 15 tables, the schema has held up well. The decisions made in `schema-decisions.md` were sound:
- UUID PKs with unique constraints work well
- The three-tier soil/water body fallback pattern is elegant
- Source tracking with `{field}_source` FKs provides good provenance
- The polymorphic trends table avoids table proliferation

## Open Questions for Strategy Discussion

1. **What's the target user experience?** The enrichment priorities depend heavily on whether we're building a wine encyclopedia, a buying guide, a cellar management tool, or a sommelier assistant. Each emphasizes different tables.

2. **External data licensing:** Scores from Wine Spectator, Parker, etc. have licensing implications. CellarTracker community data may be more accessible. What's the data acquisition strategy?

3. **Real-time vs. batch enrichment:** Do we enrich everything upfront, or build an on-demand pipeline that enriches wines when they're first viewed?

4. **Frontend timeline:** Is there a frontend being built in parallel? If so, what pages/features are coming first? That should drive enrichment priorities.

5. **Budget for external APIs:** Weather APIs (Open-Meteo = free), geocoding (Google = paid), wine data (varies). What's the comfort level?
